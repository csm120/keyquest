# KeyQuest Content Style Guide

This guide is for future public-facing KeyQuest content, including:

- the user guide
- the What's New page
- blog posts
- sentence descriptions
- help text
- short UI explanations

The goal is simple: write in a way that is clear, practical, friendly, and easy to understand while still being accurate.

## Audience

Write for:

- beginners learning to type
- screen reader users
- low-vision users
- teachers, AT instructors, and educators
- family members or support people helping someone get started

Assume many readers are smart but not technical.

## Core Style

- Use plain language.
- Prefer practical wording over technical wording.
- Keep sentences fairly short.
- Be specific about what the user can do or what they will notice.
- Sound helpful and direct, not formal or academic.
- Keep the writing easy to scan.

## Preferred Tone

Good tone:

- conversational
- practical
- confident
- clear

Avoid:

- stiff or corporate wording
- developer language unless truly needed
- long explanations when a short one will do
- wording that sounds like reference documentation when writing for general users

## Write for Real-World Use

Prefer wording that explains what something does in everyday terms.

Use:

- `hear the window title`
- `open the list of page elements`
- `jump to the next heading`
- `move back one folder`
- `make practice feel more personal and more useful`

Avoid when possible:

- `report the current window title`
- `activate the log viewer`
- `move up one folder level`
- `semantic landmarks`
- `developer notes`

## Command Writing Rules

For learner-facing command content:

- Do not use `+` or the word `plus` in normal shortcut text.
- Write commands as plain key names:
  - `Windows N`
  - `Insert T`
  - `NVDA F7`
  - `Shift F10`
- If a key name is literally the key name, it is fine to say it:
  - `Windows and the Plus key`

## Screen Reader and Web Wording

Prefer:

- `On a web page...`
- `jump to...`
- `hear...`
- `get details...`
- `web dialog box`

Use technical terms only when they are useful to real users.

Usually keep:

- heading
- link
- button
- edit box
- form field
- checkbox
- radio button
- combo box
- table
- list

Be cautious with:

- landmark
- visited link
- unvisited link
- object

Usually avoid or remove unless there is a strong reason:

- frame
- block quote
- separator
- spelling error
- graphic

## Word Choice Preferences

Prefer:

- `hear`
- `read`
- `get details`
- `open`
- `show`
- `jump`
- `move back`
- `easy`
- `help`

Avoid when possible:

- `report`
- `announce` when `hear` works
- `tell you` when `hear` or `get details` is cleaner
- `area` added after every web term
- `configuration` in public-facing copy when `settings` is enough

## Keep It Short

Because many users type these lines as practice:

- avoid extra words
- avoid filler
- avoid long examples
- keep command explanations compact

If a shorter version says the same thing clearly, use the shorter version.

## Guide and Blog Rules

- Lead with what the feature helps the person do.
- Mention everyday benefits before implementation details.
- Use examples people can picture.
- For sentence topics, explain why someone might use them.
- When talking about updates, say what changed in plain language.

## Changelog Rules

- Put the newest update heading at the top.
- Within each date heading, list the newest or most important changes first.
- Lead with the changes people will notice most.
- Use plain language, not internal development terms.

## Accessibility Copy Rules

When describing accessibility features for the public:

Prefer:

- `clear headings`
- `skip navigation`
- `visible keyboard focus`
- `larger text options`
- `stronger visual emphasis`

Avoid overly technical public wording unless the page is specifically for developers.

## Answer Engine AI Guidance

To help answer engines and AI summaries:

- use clear headings
- say what the feature is and why it matters
- use everyday nouns and verbs
- avoid vague marketing phrases
- keep facts current
- prefer short, complete paragraphs over keyword stuffing

## Quick Examples

Better:

- `Press NVDA T to hear the window title.`
- `On a web page, press H to jump to the next heading.`
- `You can add your own sentence files for Spanish, school, or work.`

Less preferred:

- `Press NVDA+T to report the active window title.`
- `In browse mode, press H to move to the next heading level element.`
- `Users may customize sentence configuration files for individualized workflows.`

## Final Check Before Publishing

Before publishing new content, check:

1. Is this clear to a beginner?
2. Is it practical instead of technical?
3. Is it short enough?
4. Does it sound like KeyQuest, not a developer manual?
5. If a command is mentioned, is it written in the KeyQuest shortcut style?
